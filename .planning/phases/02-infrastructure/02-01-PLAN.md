  <files>sanity.config.ts</files>
  <action>
Refactor sanity.config.ts to use an array of workspace configurations passed to defineConfig. This is the standard Sanity v3+ approach for multi-dataset support.

1. Extract the shared config (theme, plugins, schema, studio components, tools, document badges/actions) into a reusable `sharedConfig` object.

2. Export `defineConfig([...])` with two workspace entries:

```typescript
export default defineConfig([
  {
    ...sharedConfig,
    name: 'production',
    title: 'Autori Mandatum',
    projectId: 'fo6n8ceo',
    dataset: 'production',
    basePath: '/production',
  },
  {
    ...sharedConfig,
    name: 'staging',
    title: 'Autori Mandatum (Staging)',
    projectId: 'fo6n8ceo',
    dataset: 'staging',
    basePath: '/staging',
  },
])
```

3. The `basePath` property is REQUIRED for multi-workspace configs — Sanity uses it to route between workspaces. Each workspace gets its own URL segment.

4. Keep ALL existing functionality intact: badges, actions, custom tools (dashboard), plugins (structureTool, visionTool, assist, media), studio components (logo, navbar, head).

5. The `sharedConfig` object should contain: theme, icon, plugins, schema, studio, tools, document (badges + actions). Everything except name, title, projectId, dataset, basePath.

6. Do NOT change any imports or remove any existing code beyond the restructuring.
